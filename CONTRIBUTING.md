# Contributing to Callcium

Callcium is maintained, not community-driven. Design decisions are made by the maintainers.

## Bug Reports

Search existing issues before opening a new one. If you find a match, add context there instead.

Open an issue with a clear reproduction.

## Feature Requests

Welcome as discussion, not as PRs. Open an issue describing the problem you're solving, the use cases that motivate it, and why it belongs in the engine.

## Pull Requests

PRs are by invitation. If you send unsolicited code, expect it to be closed without merge. This isn't personal — it's how we keep scope tight and the codebase auditable. Every PR must have a human author who understands and stands behind the change.

If you want to contribute code, start with an issue. If we agree it belongs, we'll ask for a PR.

## Security Vulnerabilities

Do not open public issues for security vulnerabilities. Use [GitHub's private vulnerability reporting](https://github.com/provencraft/callcium/security/advisories/new) instead.
